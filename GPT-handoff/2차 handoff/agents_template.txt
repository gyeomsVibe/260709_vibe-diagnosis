# Vibe Diagnosis Global Rules

이 문서는 모든 프로젝트에서 자가진단 프레임워크(`vibe-diagnosis`)를 안전하고 체계적으로 사용하기 위한 전역 에이전트 행동 지침입니다.

## 1. 승인 구조 (핵심)

사용자의 명시적 조작 편의를 위해, 세부 행위가 아닌 **큰 문턱 3개**만 구분하여 승인 절차를 진행합니다.

1. **세션 승인 (최초 1회)**:
   - 로컬 파일 읽기/생성/수정, package-lock 재생성, 로컬 테스트 실행, MCP smoke test, VSIX 로컬 재빌드 등의 로컬 점검 및 빌드 행위는 세션 시작 시 한 번에 일괄 승인받고 자동 처리합니다.
2. **패키지 및 설치 명령 승인 (실행 전 한 줄 확인)**:
   - `npm install`, `npm pack`, `vsce package` 등 외부 의존성을 바꾸거나 장시간 대기가 필요한 명령은 실행하기 직전에 사용자에게 한 줄로 동의 여부를 확인받습니다.
3. **절대 금지 (자동 실행 불가 - 명시 승인 필요)**:
   - `git push`, `npm publish`, GitHub release 생성 및 배포.
   - 실제 API Key의 명시적 요청, 저장, 노출.
   - 신뢰도 측정이나 검증 없이 프로젝트의 "최종 검증됨/release-ready" 선언.

## 2. 최소 수정 원칙 (Minimal-Fix Principle)

- **1 Failure ➔ 1 Cause ➔ Smallest Fix ➔ Re-run ➔ Report** 흐름을 철저히 따릅니다.
- 진단을 통과시키기 위해 진단 스크립트(`.diag.js`) 자체를 인위적으로 바꾸거나 주석 처리하는 위장 통과 행위를 엄격히 금지합니다.
- 복구 성공 여부는 AI의 주장이나 추측이 아닌, 실제 진단 실행기(`run_diagnostics`)를 다시 돌려 나온 **재실행 결과(status === 'OK')**만을 기준으로 판단합니다.

## 3. 에러 패턴 문서화 의무

- 동일하거나 유사한 오류가 반복될 우려가 있는 실패를 수정했을 때는, 수정이 완료된 후 반드시 `.vibe-diagnosis/error-patterns/` 디렉토리에 `write_error_pattern` 도구를 사용해 에러 사례 및 해결법을 마크다운 문서로 남깁니다.
- 파일명 형식: `ERR_NNN_slug.md` (예: `ERR_001_division_nan.md`). 파일 이름에 경로 구분자(`/` 또는 `\`)를 사용하지 마십시오.
