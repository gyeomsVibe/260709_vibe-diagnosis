## **0단계. 지금 우리가 하는 일**

`vibe-diagnosis`는 CLI, VS Code Extension, MCP Server가 모두 같은 진단 코어를 쓰는 구조입니다. 업로드된 분석 문서에는 CLI, VS Code, MCP가 client layer이고, `runner.js`, `schema.js`, `init.js`, `config-manager.js`, `reporter.js`가 core engine으로 정리되어 있습니다.

쉽게 말하면:

CLI \= 사람이 터미널에서 누르는 버튼  
VS Code Extension \= 에디터 안의 버튼  
MCP Server \= Antigravity / Claude Code 같은 AI가 누르는 버튼  
Core Engine \= 실제 검사기  
Dashboard \= 결과판  
Repairer \= AI 교정 실행기

이번 단계는 코드를 막 고치는 단계가 아니라, **무엇부터 고쳐야 재작업이 줄어드는지 순서를 정하는 단계**입니다.

## **1단계. 고밀도 패치 큐**

### **P0. 증거 상태 분리**

가장 먼저 해야 할 일은 이것입니다.

원본 ZIP 상태  
현재 업로드 ZIP 상태  
GitHub 원격 상태  
VSIX 산출물 상태  
MCP node\_modules 상태

이 5개를 절대 한 덩어리로 보면 안 됩니다.

초보자식으로 말하면, **냉장고 안 재료, 요리된 음식, 배달 앱 사진을 한꺼번에 “같은 음식”이라고 부르면 안 되는 것**과 같습니다.

현재 기준에서 확실히 분리해야 할 것:

1\. 업로드 ZIP 내부 소스  
2\. ZIP 내부 .git commit  
3\. ZIP 내부 node\_modules  
4\. ZIP 내부 VSIX 파일  
5\. GitHub 웹/원격 최신 상태

이 단계의 산출물은 `STATE_BOUNDARY.md` 하나면 됩니다.

### **P1. MCP 실행 경로 정리**

MCP Server는 AI 코딩 도구가 쓸 수 있는 도구 6개를 제공합니다: `run_diagnostics`, `init_diagnostics`, `list_diagnostics`, `read_error_pattern`, `write_error_pattern`, `open_dashboard`입니다.

즉, “원터치 점검”의 최소 순서는 이미 정해져 있습니다.

init\_diagnostics  
→ list\_diagnostics  
→ run\_diagnostics  
→ 실패 확인  
→ 필요한 코드 수정  
→ run\_diagnostics 재실행  
→ write\_error\_pattern  
→ open\_dashboard

여기서 P1의 목표는 **AI가 MCP를 호출할 때 항상 같은 순서로 움직이게 만드는 것**입니다.

필요 산출물:

.claude/skills/vibe-check/SKILL.md  
GEMINI.md 또는 .gemini/settings.json 정합성 확인  
MCP 실행 예시 문장

---

### **P2. 패키지/락파일 drift 정리**

전수분석에서 확인된 drift는 다음 패치 묶음으로 처리합니다.

mcp-server/package.json  
mcp-server/package-lock.json  
mcp-server/node\_modules 포함 여부  
vscode-extension/package.json  
vscode-extension/package-lock.json  
vscode-extension/\*.vsix

초보자식으로 말하면:

package.json \= 주문서  
package-lock.json \= 실제 배송 기록  
node\_modules \= 배송된 물건  
VSIX \= 포장 완료된 상품

주문서와 배송 기록과 포장 상품이 서로 다르면, AI가 “어느 게 진짜야?” 하고 헷갈립니다.

P2의 목표는 **소스 기준, 락파일 기준, 산출물 기준을 한 줄로 맞추는 것**입니다.

---

### **P3. CLI 사용법 정리**

분석 문서 기준 CLI는 `init`, `run`, `dashboard`, `config`, `repair` 명령을 제공합니다. 또한 진단 에러가 있으면 process exit code `1`을 반환해서 CI/CD와 연결할 수 있는 구조로 설명되어 있습니다.

여기서 중요한 것은 사용자 문서입니다.

틀린 이해:  
vibe-diag \--cwd examples/calculator

정확한 형태:  
vibe-diag run \--cwd examples/calculator

P3 목표:

README 예시 정리  
CLI help 문구 정리  
테스트 명령 예시 정리

---

### **P4. Repairer 동작 경계 문서화**

분석 문서 기준 `repairer.js`는 실패 로그, 진단 소스, 프로젝트 구조, `package.json`, 과거 error-patterns를 모아 AI에게 전달하고, AI 응답으로 받은 전체 파일 내용을 로컬 파일에 반영하며, 반영 전 `.bak` 백업을 생성합니다. 이후 해당 진단을 재실행해 `status === 'OK'` 여부를 판별합니다.

초보자식으로 말하면:

repairer는 “부분 수정 펜”이 아니라  
“파일 전체를 다시 써주는 자동 교정 선생님”에 가깝다.

따라서 P4에서는 기능을 바꾸기 전에, 먼저 문서에 이렇게 못 박아야 합니다.

Auto Repair는 전체 파일 치환 방식이다.  
수정 전 .bak 백업을 만든다.  
수정 후 진단을 다시 실행한다.  
OK 판정은 재실행 결과 기준이다.

안전/보안 관련 내용은 새 의견을 붙이지 않고, 자료에 기록된 사실만 유지합니다. 업로드 문서에는 `.vibe-diagnosis/config.json`에 `apiKey`가 플레인 텍스트로 저장되며, 초기화 시 `.gitignore` 자동 삽입 메커니즘이 언급되어 있습니다.

## **2단계. REDTEAM 관점 결함 압축**

여기서 REDTEAM은 “비판하기”가 아니라 **나중에 헷갈릴 지점을 미리 제거하는 것**입니다.

### **숨은 혼동 1: MCP와 Repairer 역할 혼동**

MCP \= AI가 누를 수 있는 버튼 묶음  
Repairer \= 실패한 파일을 AI로 고치는 로직  
Runner \= 실제 진단 실행기  
Dashboard \= 결과를 보는 화면

분석 문서의 구조상 MCP는 `Runner`와 `Init`을 도구로 연결하고, Dashboard는 `Runner`와 `Repairer`를 연결합니다.

패치 방향:

문서에서 “MCP가 고친다”라고 쓰지 말고  
“MCP가 진단을 실행하고, 교정 흐름을 호출한다”로 정리한다.

---

### **숨은 혼동 2: VS Code Extension과 VSIX 혼동**

vscode-extension/src/extension.js \= 현재 소스  
.vsix \= 포장된 배포 산출물

둘이 다르면 사용자는 “왜 소스 수정했는데 설치 결과가 안 바뀌지?”라고 느낍니다.

패치 방향:

VSIX를 source 기준으로 재생성하거나  
ZIP/레포에서 VSIX를 산출물로 분리한다.

---

### **숨은 혼동 3: 원터치 자동화의 범위 과장**

원터치라고 해도 실제 내부는 여러 단계입니다.

프로젝트 확인  
→ 초기화 여부 확인  
→ 진단 목록 확인  
→ 진단 실행  
→ 실패 원인 확인  
→ 코드 수정  
→ 재진단  
→ 결과 설명

패치 방향:

“원터치”는 사용자 입력이 한 번이라는 뜻이지,  
내부 검증 단계가 하나라는 뜻이 아니라고 문서에 적는다.
