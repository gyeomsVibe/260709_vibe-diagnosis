## 1. Opportunity Brief — 현재 파이프라인의 MIA 관점 결함 진단

현재 흐름: `발견(discover) → 실행(run) → OK/WARNING/ERROR → AI/로컬 처방 1개 → 승인 → 전체 파일 치환 → 해당 진단 1개 재실행 → 성공/실패`

| # | MIA 원칙 | 현재 상태 | 결함 (왜 문제인가) |
|---|---|---|---|
| D1 | Evidence Gate (증거 분리) | 진단 결과 = 상태+문장 1줄 | 관찰 사실/추정/재현 조건이 뒤섞임. 처방 엔진이 문자열 매칭에 의존(이번 Node 22 Schema violation 오검이 실증 사례) |
| D2 | 재현성 검증 | 1회 실행으로 확정 | 간헐 실패(flaky: 네트워크·타이밍)와 진짜 결함을 구분 못함 → 오진·과잉 치료 |
| D3 | 원인 후보 좁히기 (2~3개) | 원인 분석 단계 자체가 없음 | 증상→처방 직행. AI가 틀린 원인을 고치면 "치료했는데 재발" 반복 |
| D4 | Candidate Portfolio (후보 2~3개) | 처방 1개 단일 경로 | 최소 수정 vs 구조 수정 비교 불가, 사용자가 선택권 없이 전부·전무만 선택 |
| D5 | Judgment Contract (must-reject) | **진단 약화 방지 가드 없음** | AI가 `.clinic.js`의 기대값을 바꿔 "가짜 완치"를 만들 수 있음 — 원칙("never weaken a diagnostic") 미집행 |
| D6 | Regression Guard (회귀 검증) | 대상 진단 1개만 재실행 | 치료가 다른 진단을 부숴도 성공 처리. 자동 롤백 없음(.bak은 수동 복구) |
| D7 | Maturity Label | 성공=OK 단일 판정 | STATIC_CANDIDATE(적용 전)와 VERIFIED_RESULT(회귀까지 통과)의 구분 없음 |
| D8 | 학습 루프 | error-patterns는 수동 문서 | 치료 성공/실패가 지식으로 축적되지 않음 → 같은 병을 매번 AI에게 다시 물어봄 |

핵심 가설: **D5(진단 약화 가드)와 D6(회귀+롤백)만 넣어도 "치료의 신뢰"가 질적으로 달라지고, D1~D3이 처방 정확도를, D8이 반복 비용을 낮춘다.**

## 2. 큰 계획 (마스터플랜) — MIA 4단계 → 파이프라인 매핑

```
MIA 모드            Vibe Clinic 코어 대응
─────────           ─────────────────────────────────────────────
기획(문제 정의)  →  [Phase 1] 정밀 진단: 재실행 재현성 + 증거 구조화 + 원인 후보
검토(대안 비교)  →  [Phase 2] 처방 포트폴리오: 후보 2~3개 + 판단 계약(약화 금지)
실행(최소 수정)  →  [Phase 3] 안전 치료: 회귀 게이트 + 자동 롤백 + 성숙도 라벨
검증(학습 루프)  →  [Phase 4] 치료 원장: 성공/실패 자동 기록 → 로컬 처방 DB 성장
                    [Phase 5] UI 노출: 신뢰도·후보 비교·회귀 결과·롤백 버튼
```

우선순위(위험 대비 효과): **P3(안전 치료) > P1(정밀 진단) > P2(포트폴리오) > P4(학습) > P5(UI)**
— 단, 구현 순서는 의존성 때문에 P1 스키마 → P3 → P2 → P4 → P5 를 권장.

## 3. 세부 계획 (Phase별 파일 단위)

### Phase 1 — 정밀 진단: Evidence Gate (예상 3~4시간)

**P1-1. 결과 스키마 확장** — [src/schema.js](../src/schema.js)
- 진단 결과에 선택 필드 추가(하위 호환 — 기존 진단 파일 무수정 동작):
```js
{
  status, details, errorMessage,            // 기존
  confidence: 'CONFIRMED'|'SUSPECTED',      // 재현성 판정 결과
  evidence: { observed: '', expected: '', reproduction: '' }, // 증거 3분리
  causeHypotheses: [{ cause, likelihood, signal }],           // 원인 후보 ≤3
}
```

**P1-2. 재현성 재검(Flaky Gate)** — [src/runner.js](../src/runner.js)
- ERROR/WARNING 진단만 **자동 1회 재실행**(총 2회): 2회 모두 실패 → `confidence:'CONFIRMED'`, 1회만 실패 → `'SUSPECTED'` + details에 "간헐 실패 의심(재현 1/2)" 표기.
- 재실행은 실패 항목 한정이므로 정상 프로젝트의 실행 시간 증가 없음.

**P1-3. 규칙 기반 원인 후보 엔진** — 신규 `src/triage.js`
- 처방 전에 증상→원인 후보 매핑(순수 로컬, AI 무관): ESM/CJS 로딩, 모듈 미설치(`Cannot find module`), 타임아웃, 네트워크(ENOTFOUND/ECONNREFUSED/429/503), 스키마 위반, 어서션 실패. 각 후보에 `signal`(감지 근거) 명시.
- 기존 repairer의 문자열 감지 로직을 이 모듈로 이관·일원화(이번 세션의 ESM 구조 신호 포함).

수용 기준: 신규 테스트 4건 — flaky 판정 2/2·1/2 케이스, ESM/모듈미설치 원인 후보 생성, 기존 27건 무회귀.

### Phase 2 — 처방 포트폴리오 + 판단 계약 (예상 3~4시간)

**P2-1. 후보 2~3개 생성** — [src/repairer.js](../src/repairer.js)
- 후보 A: 로컬 룰 처방(triage 후보 기반, 즉시·무AI) / 후보 B: AI 최소 수정(diff 최소화 지시) / 후보 C(선택): AI 근본 수정. 각 후보에 `{ scope: 파일 수·라인 수, reversibility: .bak 복원 가능 여부, risk: 진단파일 접촉 여부 }` 산출.
- API: `/api/repair/propose`가 `candidates[]` 배열 반환(기존 단일 필드는 candidates[0]으로 하위 호환 유지).

**P2-2. 판단 계약 — 진단 약화 가드 (must-reject)** ★ 최우선 안전장치
- 처방이 `.clinic.js|.cjs`를 수정·삭제하는 경우: **원인 후보가 "진단 파일 자체의 결함"(로딩 오류·스키마 위반)일 때만 허용**, 그 외에는 `BLOCKED_WEAKENING` 사유로 자동 거부 + UI 경고. 기대값·어서션 완화 패턴(`toFixed 조작, 조건 삭제`)은 diff 검사로 탐지.

수용 기준: 진단 약화 시도 모의 제안이 차단되는 테스트, ESM 변환(합법적 진단 파일 수리)은 통과하는 테스트.

### Phase 3 — 안전 치료: 회귀 게이트 + 자동 롤백 (예상 2~3시간) ★ 효과 최대

**P3-1. 회귀 게이트** — [src/repairer.js](../src/repairer.js) `applyRepairProposal`
- 적용 후 **전체 진단 재실행**(현재도 runDiagnostics 전체 실행이므로 비용 동일). 판정을 2축으로: ① 대상 진단 OK? ② 이전에 OK였던 다른 진단이 실패로 전환됐는가(회귀)?

**P3-2. 자동 롤백**
- 회귀 발생 또는 대상 미치유 시 `.bak`에서 **자동 복원 + 재확인**, 결과를 `ROLLED_BACK` 상태로 보고. (되돌리기 경로가 코드로 보장됨)

**P3-3. 성숙도 라벨**
- 제안: `STATIC_CANDIDATE` → 적용+대상 OK: `APPLIED` → 회귀 0 확인: `VERIFIED_RESULT` → 회귀/실패: `ROLLED_BACK`. API 응답·UI 배지에 노출.

수용 기준: "치료가 타 진단을 부수는" 모의 시나리오에서 자동 롤백 + 원상복구 해시 일치 테스트.

### Phase 4 — 치료 원장(학습 루프) (예상 2시간)

- 신규 `.vibe-clinic/treatment-ledger.json`: `{ 일시, diagId, 원인 후보, 선택 처방, 결과 라벨, 소요 }` 자동 기록(append-only).
- `VERIFIED_RESULT`로 끝난 로컬 처방 패턴은 error-patterns `.md` 자동 생성(현재 수동 문서의 자동화) → triage가 다음 진단 때 이 원장을 1순위 참조 = **같은 병 재발 시 AI 호출 없이 즉시 처방**.

### Phase 5 — UI 노출 (예상 2~3시간)

- 카드: confidence 배지(확진/의심), 원인 후보 목록. 처방 모달: 후보 비교 탭(범위·위험·되돌리기), 약화 차단 경고. 결과: VERIFIED_RESULT/ROLLED_BACK 배지 + 수동 롤백 버튼(`.bak` 존재 시).

## 5. 결정 게이트

- 권고: **Go — P1(스키마·flaky·triage) → P3(회귀+롤백) 순으로 착수.** 이 두 단계만으로 "정밀 진단·안전 치료"의 골격이 완성되고, P2·P4·P5는 각각 독립적으로 얹을 수 있다.
- 각 Phase 완료 시 6분류 보고 + 다음 Phase 착수 여부 재승인. push는 매번 별도 승인.
- 재검토 조건: P1 후 실행 시간이 체감 저하되면 재검 횟수·범위 조정.
