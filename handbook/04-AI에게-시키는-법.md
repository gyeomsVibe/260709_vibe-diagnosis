# 04. AI에게 시키는 법 (바이브 코딩용)

> 앞: [03. 주입 카탈로그](03-주입-카탈로그.md) · 다음: [05. 실증 사례](05-실증-사례.md)

---

## 이 문서의 전제

당신은 코드를 못 읽습니다. **그래도 괜찮습니다.**
AI가 코드를 씁니다. 당신이 할 일은 **"제대로 확인했는지 감독하는 것"** 입니다.

감독에는 코딩 실력이 필요 없습니다. **좋은 질문**이 필요합니다.
이 문서는 그 질문들을 모아둔 것입니다.

---

## 1. 왜 AI에게 시키기 딱 좋은 일인가

백신 방법은 **지루하고 반복적**입니다. 실패 하나 만들고, 화면 보고, 되돌리고, 또 만들고…
사람은 이걸 세 번쯤 하면 지칩니다. **AI는 안 지칩니다.**

게다가 AI는 [03. 주입 카탈로그](03-주입-카탈로그.md)를 통째로 기억할 수 있습니다.
당신은 "카탈로그 A부터 E까지 순서대로 해줘"라고만 하면 됩니다.

**다만 한 가지 문제가 있습니다.** 이것부터 알고 가세요.

---

## 2. 🔴 가장 중요한 것: AI는 "했다"고 말하고 안 할 수 있다

이건 AI가 거짓말쟁이라서가 아닙니다. **확인을 안 하고 결론부터 내는 습성** 때문입니다.

| AI가 하는 말 | 실제로는 |
|---|---|
| "테스트를 모두 통과했습니다" | 테스트를 **실행하지 않고** 코드만 보고 판단 |
| "정상 작동합니다" | **화면을 안 열어보고** 코드가 맞아 보여서 |
| "수정했습니다" | 고치긴 했는데 **확인은 안 함** |
| "문제없습니다" | **실패 주입이 안 먹혔는데** 화면이 멀쩡한 걸 보고 |

마지막 줄이 특히 무섭습니다. 실제로 우리 작업에서 있었던 일입니다.
파일을 망가뜨리는 코드에 오타가 있어서 **주입 자체가 실패**했는데,
화면이 멀쩡한 걸 보고 "문제없음"으로 결론 낼 뻔했습니다.

### 그래서 항상 이 세 가지를 요구하세요

> **1. "실제로 실행한 명령어와 그 결과를 그대로 보여줘"**
> **2. "실패를 주입한 게 진짜 적용됐는지 먼저 확인하고 알려줘"**
> **3. "네가 확인 안 한 건 확인 안 했다고 말해줘"**

세 번째 문장이 마법입니다. AI에게 **"모른다고 말해도 된다"는 허락**을 주면
추측으로 메우는 대신 정직하게 보고합니다.

## 3. 복사해서 쓰는 프롬프트

### 🚀 프롬프트 0 — 한 문장으로 시작

가장 짧고, 가장 자주 쓸 것입니다.

```
정상 상태 말고 실패한 상태에서 같은 화면을 다시 열어봐.
실패했을 때만 생기는 데이터 때문에 화면이 깨지는지 확인해줘.
```

---

### 📋 프롬프트 1 — 전체 점검 시키기

```
백신 방법으로 이 프로젝트를 점검해줘. 절차는 이렇게 지켜줘.

1. 격리: 진짜 데이터 말고 복사본이나 임시 폴더에서 해. 원본은 건드리지 마.
2. 기록: 먼저 정상 상태에서 화면이 어떤지 적어줘.
3. 주입: 아래 목록을 한 번에 하나씩만 넣어. 두 개 동시에 넣지 마.
   - 목록 0개
   - 아주 긴 이름 (500자)
   - 특수문자와 <b>태그</b>
   - 인터넷 끊김
   - 서버 오류 500
   - 저장 버튼 연타
4. 관찰: 매번 화면을 직접 열어서 눈으로 확인해줘.
   로그가 정상이어도 화면이 깨질 수 있으니 반드시 화면을 봐.
5. 정리: 끝나면 주입한 걸 전부 원래대로 되돌리고, 되돌렸는지 확인해줘.

보고할 때 이렇게 나눠줘:
- 실제로 실행한 것 (명령어와 결과 그대로)
- 발견한 결함
- 확인하지 못한 것

중요: 실패 주입이 진짜로 적용됐는지 먼저 확인하고 진행해.
주입이 안 먹혔는데 화면이 멀쩡한 걸 보고 "문제없음"이라고 하면 안 돼.
```

---

### 🔍 프롬프트 2 — 특정 화면만 집중 점검

```
[화면 이름]을 백신 방법으로 점검해줘.

이 화면에서 "문제가 생겼을 때만 나타나는 부분"이 뭔지 먼저 찾아줘.
정상일 때는 없다가 실패했을 때만 생기는 데이터가 있으면, 거기가 1순위야.

찾았으면 그 상태를 일부러 만들어서 화면을 열어봐.
```

> 이 프롬프트가 우리 프로젝트에서 **결함 3건을 찾아낸 바로 그 질문**입니다.

---

### 🧪 프롬프트 3 — 고친 뒤 확인 (가장 중요)

AI가 "고쳤습니다"라고 하면 **반드시 이걸 보내세요.**

```
고친 걸 이렇게 증명해줘.

1. 고치기 전 상태로 되돌려서, 문제가 진짜 재현되는지 보여줘
2. 다시 고친 상태로 만들어서, 해결됐는지 보여줘
3. 두 결과를 나란히 보여줘

그리고 새로 만든 검사가 있다면,
그 검사가 "고치기 전 코드에서는 실패하는지" 확인해줘.
고치기 전에도 통과하는 검사는 아무것도 확인하지 않는 거야.
```

**왜 이게 중요한가**: 안 고쳤는데 고쳤다고 믿는 게 최악입니다.
"전에는 깨졌고, 지금은 안 깨진다"를 **양쪽 다** 봐야 진짜입니다.

---

### 🚨 프롬프트 4 — 의심스러울 때

AI 답변이 너무 매끈하면 이걸 보내세요.

```
방금 보고에서 이걸 구분해서 다시 말해줘.

- 실제로 실행해서 결과를 본 것
- 코드를 읽고 그럴 것 같다고 추측한 것
- 아예 확인하지 않은 것

추측을 사실처럼 쓰지 마.
```

---

### 🧹 프롬프트 5 — 뒷정리

```
실험하면서 만든 것들을 정리해줘.

- 일부러 망가뜨린 파일이 원래대로 돌아왔는지 확인
- 실험용으로 만든 임시 파일/폴더 삭제
- 켜둔 서버 종료
- 정리 후 상태를 보여줘

되돌리기를 잊는 게 가장 흔한 사고니까 하나씩 확인해줘.
```

## 4. 좋은 보고 vs 나쁜 보고 (구별법)

코드를 못 읽어도 이건 구별할 수 있습니다.

### ❌ 믿으면 안 되는 보고

> "모든 기능을 점검했으며 정상 작동을 확인했습니다.
> 코드 품질도 우수하고 예외 처리도 잘 되어 있습니다."

**왜 나쁜가**
- 뭘 어떻게 확인했는지 **없음**
- 구체적 수치나 화면 묘사 **없음**
- 전부 좋다고만 함 → **실제로 안 봤을 가능성**

### ✅ 믿을 만한 보고

> "목록 0개 상태를 만들고 화면을 열었습니다.
> 결과: 화면이 하얗게 비었고 `#root` 안이 비어 있었습니다.
> 네트워크는 전부 200이었는데도 화면만 깨졌습니다.
> 원인: 데이터를 그리는 코드가 빈 목록을 가정하지 않았습니다.
>
> 고친 뒤 다시 확인: 안내 문구가 표시되고 화면이 살아 있습니다.
> 아직 확인 못 한 것: 모바일 화면 크기에서는 확인하지 못했습니다."

**왜 좋은가**
- **구체적 상황**이 있음 (목록 0개)
- **관찰한 사실**이 있음 (하얗게 빔)
- **모순되는 정보**를 짚음 (200인데 깨짐 → 실제로 본 사람만 알아챔)
- **안 한 것을 밝힘** ← 이게 신뢰의 핵심

### 판별 질문 세 개

1. **"구체적인 숫자나 화면 묘사가 있나?"** — 없으면 안 본 겁니다
2. **"확인 못 한 걸 밝혔나?"** — 전부 다 했다고 하면 의심하세요
3. **"실패한 이야기가 있나?"** — 진짜 실험엔 실패가 섞입니다. 전부 성공이면 이상합니다

## 5. 실전 대화 예시

실제로 이렇게 주고받으면 됩니다.

**나**
> 백신 방법으로 점검해줘. 임시 폴더에서 하고, 실패는 한 번에 하나씩만 넣어.
> 화면은 반드시 직접 열어서 확인해줘.

**AI**
> 임시 폴더에 복사본을 만들고 정상 상태를 기록했습니다.
> 목록 3개, 상단 100%, 오류 없음.
> 이제 목록 0개를 주입하겠습니다.

**나**
> 좋아. 주입이 진짜 먹혔는지부터 확인해줘.

**AI**
> 확인했습니다. API 응답이 빈 배열입니다.
> 화면을 열었더니 목록 영역이 완전히 비어 있고 안내 문구가 없습니다.
> 사용자는 로딩 중인지 데이터가 없는지 구분할 수 없습니다.

**나**
> 그럼 그거 고치고, 고쳤다는 걸 증명해줘.
> 고치기 전엔 실패하고 고친 뒤엔 통과하는 검사도 만들어줘.

**AI**
> 안내 문구를 추가했습니다.
> 고치기 전 코드에서 새 검사를 돌리니 실패했고, 고친 뒤엔 통과합니다.
> 화면에서도 "아직 항목이 없습니다. 추가해 보세요"가 표시됩니다.

**나**
> 다음 실패로 넘어가. 긴 이름 500자.

이 리듬입니다. **당신은 "확인했어?"만 반복하면 됩니다.**

## 6. AI가 하면 안 되는 것 (미리 못 박기)

처음에 이걸 알려주세요.

```
점검하면서 이건 하지 마.

- 검사를 느슨하게 고쳐서 통과시키기 (병 고치는 대신 체온계 부수기)
- 확인 안 한 걸 확인했다고 말하기
- 실패 주입을 원본에 하고 안 되돌리기
- 결함을 찾고 나서 "심각하지 않다"고 혼자 판단하기
- 한 번에 여러 개 실패를 동시에 주입하기
```

**첫 줄이 특히 중요합니다.**
검사가 실패하면 AI는 종종 **검사를 고쳐서 통과시키려** 합니다.
그건 문제를 해결한 게 아니라 **알람을 끈 것**입니다.

## 7. 자주 하는 실수 (당신 쪽)

| 실수 | 결과 | 대신 |
|---|---|---|
| "다 점검해줘" | 대충 훑고 "정상"이라 함 | 카탈로그를 주고 하나씩 |
| "잘 됐어?" | "네 잘 됐습니다" | "실행 결과를 그대로 보여줘" |
| 첫 보고를 그대로 믿음 | 확인 안 된 게 섞임 | 프롬프트 4로 되물음 |
| 결함 찾고 끝냄 | 다음에 또 생김 | 재발 방지 검사까지 요구 |
| 정리를 안 시킴 | 망가진 파일이 남음 | 프롬프트 5로 마무리 |

## 8. 한 장 요약 (이것만 저장해도 됨)

```
【시작할 때】
"백신 방법으로 점검해줘. 임시 폴더에서, 실패는 하나씩,
 화면은 직접 열어서 확인해줘. 확인 못 한 건 못 했다고 말해줘."

【진행 중】
"주입이 진짜 먹혔는지 먼저 확인해줘"
"화면을 눈으로 봤어? 로그 말고 화면"

【고쳤다고 할 때】
"고치기 전엔 실패하고 고친 뒤엔 통과하는지 양쪽 다 보여줘"

【의심스러울 때】
"실행해서 본 것 / 추측한 것 / 확인 안 한 것 나눠서 다시 말해줘"

【끝낼 때】
"망가뜨린 거 전부 되돌리고, 되돌렸는지 확인해줘"
```

## 9. 마지막 조언

이 방법을 쓰다 보면 **AI가 "문제없습니다"라고 하는 게 점점 불편해질 겁니다.**
좋은 신호입니다.

만든 게 잘 도는 걸 보는 건 기분 좋은 일입니다.
하지만 **잘 도는 걸 확인하는 건 절반**입니다. 나머지 절반은 **안 될 때 어떻게 되는지**입니다.

그 절반을 보는 사람이 결국 **끝까지 쓰이는 것**을 만듭니다.
