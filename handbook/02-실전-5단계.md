# 02. 실전 5단계 — 이대로 따라 하세요

> 앞: [01. 백신 원리](01-백신-원리.md) · 다음: [03. 주입 카탈로그](03-주입-카탈로그.md)

---

## 전체 그림

```
① 격리        복제본을 만든다              "환자 말고 배양접시에서"
     ↓
② 기록        멀쩡할 때를 적어둔다          "평소 체온을 알아야 열을 안다"
     ↓
③ 주입   ←┐   실패를 하나 만든다            "약한 병원체를 넣는다"
     ↓    │
④ 관찰    │   화면을 열어 눈으로 본다        "증상을 본다"
     ↓    │
   결함?  │
   ├ 아니오 ┘  (다음 실패로)
   └ 예
     ↓
⑤ 항체        고치고 재발 방지 검사를 남긴다  "면역을 남긴다"
```

③④는 **반복**합니다. 한 번에 하나씩, 카탈로그를 따라 내려가면서요.

---

## ① 격리 — 복제본을 만든다

### 왜

실패를 주입한다는 건 **일부러 망가뜨린다**는 뜻입니다.
진짜 데이터나 남이 쓰는 서비스에 하면 그건 실험이 아니라 **사고**입니다.

카오스 엔지니어링에서 가장 강조하는 원칙이 **"폭발 반경 최소화"** 입니다.
공식 원칙 문서는 이걸 실험자의 **"책임이자 의무"** 라고 표현합니다.

### 어떻게

셋 중 하나면 됩니다. 위로 갈수록 안전합니다.

1. **임시 폴더에 복사본** — 가장 안전. 끝나면 폴더째 삭제
2. **테스트용 계정/프로젝트** — 진짜와 분리된 공간
3. **되돌릴 준비를 하고 원본에서** — 최후의 수단. 반드시 원본 백업 먼저

### 흔한 실수

- ❌ "잠깐만 바꿨다가 되돌리면 되지" → **되돌리는 걸 잊습니다.** 반드시 잊습니다.
- ❌ 원본을 고쳐놓고 다른 일을 시작함 → 망가진 상태가 그대로 남습니다.

### 체크리스트

- [ ] 지금 망가뜨릴 대상이 **복제본**인가?
- [ ] 원본은 안전한 곳에 있나?
- [ ] 실험이 끝나면 **어떻게 되돌릴지** 미리 정했나?

## ② 기록 — 멀쩡할 때를 적어둔다

### 왜

열이 나는지 알려면 **평소 체온**을 알아야 합니다.
"이게 원래 이런 건지, 내가 망가뜨려서 이런 건지" 헷갈리면 실험이 무의미해집니다.

카오스 엔지니어링에서는 이걸 **정상 상태 가설(Steady State Hypothesis)** 이라고 부릅니다.
핵심은 **"내부가 아니라 겉으로 측정 가능한 결과"** 를 본다는 것입니다.
코드가 어떻게 생겼는지 말고, **화면에 뭐가 보이는지**를 적으세요.

### 어떻게

주입하기 **전에** 정상 상태를 적습니다. 사진을 찍어두면 더 좋습니다.

```
[정상 상태 기록]
- 목록에 항목이 3개 보인다
- 상단 숫자가 "100%"다
- 버튼이 3개 있고 전부 눌린다
- 오류 메시지 없음
```

### 흔한 실수

- ❌ 기록 없이 바로 주입 → 나중에 "원래 이랬나?" 하고 헷갈립니다
- ❌ "코드가 이렇게 되어 있으니까"로 기록 → **눈에 보이는 것**을 적으세요

### 체크리스트

- [ ] 정상일 때 화면을 **직접 열어봤나?**
- [ ] 보이는 것을 문장으로 적었나?
- [ ] 스크린샷을 남겼나? (선택이지만 강력 추천)

## ③ 주입 — 실패를 하나 만든다

### 왜

여기가 백신 접종입니다. **약화된 병원체 = 통제된 실패.**

### 어떻게

[03. 주입 카탈로그](03-주입-카탈로그.md)에서 하나 고르세요. 예를 들면:

- 목록을 **0개**로 만든다
- 이름에 **이모지와 `<태그>`** 를 넣는다
- 응답을 **30초 지연**시킨다
- 인터넷을 **끊는다**

### 🔴 가장 중요한 규칙: 한 번에 하나만

두 개를 동시에 넣으면 **뭐가 원인인지 알 수 없습니다.**
화면이 깨졌는데 빈 목록 때문인지 긴 이름 때문인지 모르면, 고칠 수도 없습니다.

```
❌ 나쁜 예: 목록 0개 + 이름에 이모지 + 응답 지연  → 동시에 세 개
✅ 좋은 예: 목록 0개 → 확인 → 되돌림 → 이모지 → 확인 → 되돌림 → ...
```

### 흔한 실수

- ❌ 너무 극단적인 주입 → "이런 일은 절대 안 생겨"라며 무시하게 됩니다. **현실에서 일어날 법한 것**부터.
- ❌ 주입해놓고 안 되돌림 → 다음 실험이 오염됩니다.

### 체크리스트

- [ ] **한 가지만** 주입했나?
- [ ] 이 실패가 현실에서 **일어날 법한** 일인가?
- [ ] 주입이 **실제로 적용됐는지** 확인했나? (안 먹혔는데 "문제없음"으로 결론 내는 게 최악)

> 실제로 겪은 일입니다. 파일을 바꾸는 코드에 오타가 있어서 **주입 자체가 실패**했는데,
> 화면이 멀쩡한 걸 보고 "괜찮네"라고 결론 낼 뻔했습니다.
> **주입이 먹혔는지부터 확인하세요.**

## ④ 관찰 — 화면을 열어 눈으로 본다

### 왜

**이 단계를 건너뛰면 앞의 모든 게 무의미합니다.**

가장 흔한 실패 방식이 이겁니다: 로그만 보고, 응답 코드만 보고 "정상"이라고 판단하는 것.

실제 사례를 그대로 옮기면 이렇습니다.

> 네트워크 요청은 **전부 200(성공)** 이었습니다. 서버 로그도 깨끗했습니다.
> 그런데 **화면은 하얗게 비어 있었습니다.**
> 데이터를 받은 뒤 화면에 그리는 단계에서 터졌기 때문입니다.

**"통신 성공 ≠ 화면 정상"** 입니다. 반드시 눈으로 보세요.

### 무엇을 볼 것인가

| 볼 것 | 나쁜 신호 |
|---|---|
| **화면 전체** | 하얗게 빔, 일부가 사라짐, 레이아웃 깨짐 |
| **메시지** | 영어 원문 에러, `undefined`, `[object Object]`, `NaN` |
| **다음 행동** | 이 화면에서 **다음에 뭘 해야 할지 알 수 있나?** |
| **되돌아가기** | 실패 후 원래 화면으로 **돌아갈 수 있나?** |
| **버튼** | 눌리는데 아무 일도 안 일어남, 계속 "처리 중…" |

### 특히 중요한 질문 두 개

1. **"사용자가 이 화면을 보고 뭘 해야 할지 알까?"**
   - "오류가 발생했습니다"만 뜨면 → 사용자는 아무것도 못 합니다
2. **"실패했는데 성공처럼 보이지 않나?"**
   - 이게 가장 위험합니다. 조용한 실패는 나중에 훨씬 크게 터집니다

### 흔한 실수

- ❌ 로그만 보고 판단 → 위 사례처럼 정반대 결론이 납니다
- ❌ 개발자 도구 콘솔만 보기 → 화면이 언마운트되면 콘솔에 아무것도 안 남기도 합니다
- ❌ 대충 보고 넘어감 → "빈 목록"과 "빈 목록 + 안내 문구"는 전혀 다릅니다

### 체크리스트

- [ ] 화면을 **직접** 열어봤나?
- [ ] 화면이 살아 있나? (하얗게 안 비었나)
- [ ] 메시지를 **사람이 읽고 이해할 수 있나?**
- [ ] 다음에 뭘 해야 할지 **화면이 알려주나?**
- [ ] 실패했는데 **성공처럼 보이지는 않나?**

## ⑤ 항체 — 고치고 재발 방지 검사를 남긴다

### 왜

여기까지 안 하면 **면역이 안 생깁니다.**
고치기만 하면 나중에 누군가(당신 포함) 같은 실수를 다시 합니다.

백신의 진짜 가치는 "이번에 안 아픈 것"이 아니라 **"다음에도 안 아픈 것"** 입니다.

### 어떻게

**세 가지를 남기세요.**

**(1) 고침** — 근본 원인 하나를 고칩니다. 증상을 가리지 마세요.

```
❌ 화면이 깨지니까 그 부분을 아예 숨긴다      → 증상 가리기
✅ 데이터 형태를 잘못 다룬 부분을 고친다        → 원인 제거
```

**(2) 재발 방지 검사** — 같은 상황을 자동으로 확인하는 검사를 추가합니다.

```
"목록이 0개일 때 안내 문구가 보인다"를 확인하는 검사
```

이게 **항체**입니다. 다음에 누가 코드를 고쳐서 이 문제를 되살리면, 검사가 잡아냅니다.

**(3) 기록** — 무엇이 왜 그랬는지 한 페이지로 남깁니다.

| 항목 | 내용 |
|---|---|
| 증상 | 겉으로 보인 것 |
| 원인 | 실제 이유 |
| 왜 못 잡았나 | **이게 제일 중요합니다** |
| 해결 | 어떻게 고쳤나 |
| 재발 방지 | 무슨 검사를 남겼나 |

**"왜 못 잡았나"** 를 꼭 적으세요. 그게 다음 실험을 어디서 할지 알려줍니다.

### 흔한 실수

- ❌ 고치고 끝 → 몇 주 뒤 똑같은 게 돌아옵니다
- ❌ 검사를 느슨하게 고쳐서 통과시킴 → **이건 자해입니다.** 병을 고친 게 아니라 체온계를 부순 겁니다
- ❌ 기록 안 함 → 같은 걸 또 조사하게 됩니다

### 체크리스트

- [ ] **원인**을 고쳤나? (증상 가리기 아님)
- [ ] 재발 방지 **검사**를 남겼나?
- [ ] 그 검사가 **고치기 전에는 실패**하는지 확인했나? ← 중요
- [ ] **"왜 못 잡았나"** 를 기록했나?

> 마지막 체크가 핵심입니다. 새 검사가 **고치기 전 코드에서도 통과**한다면,
> 그 검사는 아무것도 확인하지 않는 것입니다. 반드시 "빨간불 → 초록불"을 확인하세요.

## 한 번에 도는 사이클 (요약표)

| 단계 | 한 일 | 끝났다는 신호 |
|---|---|---|
| ① 격리 | 복제본 준비 | 원본이 안전하다 |
| ② 기록 | 정상 상태 기록 | 정상일 때 화면을 문장으로 적었다 |
| ③ 주입 | 실패 1개 주입 | **주입이 먹혔음을 확인**했다 |
| ④ 관찰 | 화면 직접 확인 | 눈으로 봤고, 결과를 적었다 |
| ⑤ 항체 | 고침+검사+기록 | 검사가 고치기 전엔 실패했다 |

한 사이클이 끝나면 **③으로 돌아가** 다음 실패를 주입합니다.
[03. 주입 카탈로그](03-주입-카탈로그.md)를 위에서부터 훑으면 됩니다.

```markdown
-

**주입 후**:
-

```
