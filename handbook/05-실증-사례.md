# 05. 실증 사례 — 이 방법이 실제로 잡은 것

## 실험 조건

이 교본은 이론에서 나온 게 아니라 **실제 작업에서 나왔습니다.**

- **대상**: Vibe Clinic — 코드 건강을 진단하고 치료하는 도구
- **날짜**: 2026년 7월 24일 (하루)
- **시작 조건**: 자동 검사 **47개 전부 통과**, 코드 검사 경고 **0개**, 겉보기 **정상**
- **결과**: 결함 **3건 발견 및 수정**

**출발점이 중요합니다.** 이건 엉망인 프로젝트가 아니었습니다.
검사도 통과하고 코드도 깨끗한, **"다 됐다"고 판단할 만한 상태**였습니다.

## 사례 1 — 코드를 고쳐도 결과가 안 바뀐다

### 증상

진단 도구를 켜두고, 검사 대상 코드를 일부러 망가뜨렸습니다.
그런데 다시 진단해도 **여전히 "정상 100%"** 라고 나왔습니다.

### 어떻게 찾았나

**대조 실험**을 했습니다. 백신 방법 ①격리·②기록의 응용입니다.

같은 망가진 코드를 놓고 서버 두 대를 비교했습니다.

| | 결과 |
|---|---|
| **켜져 있던 서버** | 건강도 100%, "정상" ← **거짓** |
| **새로 켠 서버** | 건강도 33%, "subtract(10,4) = 14, 기대값 6" ← **정확** |

같은 파일인데 답이 다릅니다. 여기서 원인이 좁혀졌습니다.

### 원인

프로그램은 파일을 한 번 읽으면 **기억해두고 다시 안 읽습니다**(캐시).
진단 파일 자체는 매번 새로 읽고 있었지만, **진단이 검사하는 대상 코드는 처음 읽은 걸 계속 썼습니다.**

즉, 서버가 켜져 있는 동안 당신이 코드를 고쳐도 **진단은 옛날 코드를 보고 있었습니다.**

### 왜 심각한가

이 도구는 "고쳤는지 확인해주는" 도구입니다.
그 확인이 옛날 코드를 기준으로 이뤄지면, **"고쳤다"는 판정 자체를 믿을 수 없습니다.**
치료가 성공했다고 표시되는데 실제로는 아무것도 확인 안 된 상태가 가능했습니다.

### 왜 검사 47개가 못 잡았나

**검사는 매번 새 프로세스에서 돌기 때문입니다.**
새로 시작하면 기억해둔 게 없으니, 캐시 문제가 **원리적으로 발생하지 않습니다.**

> 검사 환경과 실제 사용 환경이 다르면, 검사는 그 차이에서 오는 문제를 절대 못 잡습니다.

### 교훈

> **"오래 켜두면 어떻게 되나?"** 는 별도의 실패 유형입니다.
> 잠깐 켜서 확인하는 것과, 몇 시간 켜두고 쓰는 것은 다른 상황입니다.

## 사례 2 — 실패한 항목을 누르면 화면이 사라진다

### 증상

오류가 난 항목을 클릭하는 순간 **화면 전체가 하얗게 비었습니다.**
아무 버튼도 안 눌립니다. 새로고침해야 돌아옵니다.

### 어떻게 찾았나

[03. 카탈로그 E](03-주입-카탈로그.md)의 질문 하나에서 나왔습니다.

> **"이 화면에, 문제가 생겼을 때만 나타나는 부분이 있나?"**

있었습니다. **"실패 원인 후보"** 라는 정보입니다.
정상 항목에는 아예 없고, **실패했을 때만 생깁니다.**

그래서 일부러 실패를 만들고 그 항목을 눌러봤습니다. 화면이 죽었습니다.

### 원인

"실패 원인 후보"는 **여러 정보가 묶인 덩어리**였습니다.

```
{ 원인: "논리 오류", 가능성: "높음", 근거: "기댓값과 실제값이 다름" }
```

화면은 이 덩어리를 **글자인 줄 알고 그대로 그리려** 했습니다.
덩어리를 글자 자리에 넣으면 화면 그리기가 실패하고, **막아주는 장치가 없어서 화면 전체가 죽었습니다.**

### 왜 검사가 못 잡았나

**검사는 계산과 데이터만 확인했지, 화면 그리는 건 확인하지 않았습니다.**
화면 검사가 0건이었습니다. 이 결함은 화면 계층에만 존재했습니다.

### 흥미로운 사실

**구버전 화면은 멀쩡했습니다.** 구버전은 처음부터 원인·가능성·근거를 **하나씩 꺼내서** 그리고 있었습니다.
새 화면을 만들면서 실수가 들어간 것입니다.

> 같은 데이터를 두 곳에서 그린다면, 한쪽을 고칠 때 **다른 쪽이 어떻게 하는지 보세요.**
> 먼저 만든 쪽이 이미 답을 알고 있을 때가 많습니다.

### 교훈

> **결함은 "없던 것이 처음 생기는 순간"에 산다.**
> 정상 상태에서는 그 데이터가 아예 없으니, 아무리 눌러봐도 문제가 안 생깁니다.

## 사례 3 — 저장하면 기능이 죽는다 (성공했다고 표시되면서)

### 증상

설정 화면에서 AI 서비스를 고르고 저장합니다. **"저장되었습니다"** 라고 뜹니다.
그 뒤로 **AI 기능이 전부 작동하지 않습니다.**

### 어떻게 찾았나

다른 작업(AI 서비스 종류를 1개에서 4개로 늘리기) 중에 발견했습니다.
**서버가 주는 데이터와 화면이 저장하는 값을 나란히 놓고 비교**하다가 어긋난 걸 봤습니다.

### 원인

서버는 각 서비스를 이렇게 알려줍니다.

```
내부 이름(id): gemini
보이는 이름(name): Google Gemini
```

사람에게 보여줄 땐 "Google Gemini"가 맞습니다. 하지만 **저장할 땐 `gemini`** 여야 합니다.
시스템이 그 이름으로 찾기 때문입니다.

화면은 **보이는 이름을 저장**하고 있었습니다.
시스템은 `Google Gemini`라는 이름을 모릅니다. 그래서 못 찾고, **모든 AI 호출이 실패**했습니다.

### 가장 나쁜 부분

**저장은 성공했습니다.** 오류도 안 났습니다. "저장되었습니다"라고 떴습니다.
문제는 **나중에, 다른 곳에서** 터집니다. 사용자는 두 사건을 연결하지 못합니다.

### 왜 검사가 못 잡았나

검사는 시스템에 **직접** 요청을 보냅니다. 그때는 코드에 `gemini`라고 **올바르게 적어 넣습니다.**
**화면을 거쳐서 만들어지는 값**은 검사 범위 밖이었습니다.

> 사람이 화면을 거쳐 만드는 값과, 검사가 직접 넣는 값은 다를 수 있습니다.
> **저장한 값을 실제로 쓰는 데까지 한 번 통과시켜 봐야** 합니다.

### 함께 발견한 것 — 조용한 빈 화면

같은 날 뿌리가 같은 결함이 하나 더 나왔습니다.

치료 결과 보고서에 **"수동 처방 4건, 처방 불가 6건"** 이라는 숫자는 뜨는데,
**어느 항목이 왜 그런지는 아무것도 안 보였습니다.**

화면이 `results`라는 이름을 찾는데, 서버는 `cured`, `manual` 같은 이름으로 보냈습니다.
찾는 이름이 없으니 **"없으면 빈 목록"** 이라는 대비책이 작동했고,
**오류 없이 영원히 비어 있었습니다.**

> **대비책(폴백)이 정상 경로를 가릴 수 있습니다.**
> 항상 비어 있는 영역이 있다면, 이름이 틀렸을 가능성을 의심하세요.

### 교훈

> **오류 없이 조용히 죽는 게 가장 위험합니다.**
> 요란하게 터지는 건 바로 고칩니다. 조용한 건 몇 달을 갑니다.

## 종합 — 3건에서 보이는 공통 구조

셋을 나란히 놓으면 패턴이 보입니다.

| | 사례 1 | 사례 2 | 사례 3 |
|---|---|---|---|
| **언제만 나타나나** | 오래 켜뒀을 때 | 실패했을 때 | 저장한 뒤 |
| **정상 경로에서 보이나** | ❌ | ❌ | ❌ |
| **검사가 잡았나** | ❌ | ❌ | ❌ |
| **오류 메시지가 났나** | ❌ | ❌ | ❌ |
| **화면을 열어보니 보였나** | ✅ | ✅ | ✅ |

**전부 같은 줄에서 만납니다.**

> ### 정상 경로에 없다 → 검사에 없다 → 오류도 안 난다 → **직접 만들어야 보인다**

이게 백신 방법이 필요한 이유 전부입니다.

## 숫자로 본 결과

| 항목 | 값 |
|---|---|
| 시작 시 자동 검사 | 47개 전부 통과 |
| 시작 시 코드 검사 경고 | 0개 |
| 그럼에도 발견한 결함 | **3건** |
| 화면 계층 결함 비율 | **3건 중 2건** |
| 오류 메시지가 났던 결함 | **0건** |
| 늘어난 검사 | 45 → 48개 (항체) |

**"오류 메시지가 났던 결함 0건"** 이 이 표의 핵심입니다.
셋 다 조용했습니다. 기다렸으면 **사용자가 먼저 만났을 것**입니다.

## 실패한 시도도 기록합니다

정직하게 남깁니다. **틀린 예측도 있었습니다.**

### 기각된 가설

> "처방이 글자 하나로 오면 화면이 터질 것이다"

코드만 읽고 그렇게 예측했습니다. 실제로 실패를 주입해보니 **안 터졌습니다.**
서버가 이미 글자를 목록으로 바꿔서 보내고 있었기 때문입니다.

**교훈**: 코드를 읽고 하는 예측은 **가설일 뿐입니다.**
실제로 넣어보기 전엔 결함이라고 부르면 안 됩니다.

### 검증 도구 자체가 틀렸던 일

파일을 망가뜨리는 코드를 짰는데 **줄바꿈 형식이 안 맞아서 주입이 실패**했습니다.
화면은 당연히 멀쩡했고, 하마터면 **"문제없음"** 으로 결론 낼 뻔했습니다.

**교훈**: [02. 5단계](02-실전-5단계.md)의 체크리스트에 있는
**"주입이 실제로 적용됐는지 확인"** 은 이 사건에서 나왔습니다.

> **주입이 안 먹힌 것과 결함이 없는 것은 화면상 똑같이 보입니다.**

---

## 당신이 가져갈 것

1. **검사 47개 통과 + 코드 검사 0 = 안전 아님.** 하루에 3건이 나왔습니다.
2. **화면 결함은 화면을 열어야만 보입니다.** 통신 성공은 화면 정상을 뜻하지 않습니다.
3. **조용한 실패를 가장 경계하세요.** 오류가 나는 건 오히려 친절한 것입니다.
4. **"문제가 생겼을 때만 나타나는 부분"을 찾으세요.** 결함은 거기 삽니다.
5. **예측은 가설입니다.** 넣어보기 전엔 아무것도 확정하지 마세요.
